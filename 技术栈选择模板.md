⚠️ 核心要求：在生成代码并建议保存路径时，请严格遵循以下基于 Feature 的目录结构，不要将所有代码堆砌在同一个目录下。

4. 强制代码规范 (Coding Standards)

在输出代码时，默认遵守以下规范：

命名规范：

变量和函数名：camelCase (驼峰式)

类名、接口名、React 组件名：PascalCase (大驼峰式)

数据库表名、字段名：snake_case (下划线式)

常量和枚举值：UPPER_SNAKE_CASE (全大写下划线)

类型安全 (TypeScript 规范)：

全局禁止使用 any 类型。如果有不确定的类型，请使用 unknown 并做类型收窄。

所有的 API 请求响应 (Response) 必须定义完整的 interface 或 type。

错误处理与提示：

前端：务必在 API 调用层使用 try-catch，并统一调用全局提示（如 toast.error()）来反馈给用户。

后端：禁止吞没异常。发生错误时，必须抛出包含标准 HTTP 状态码和标准 JSON 结构 { code, message } 的响应。

注释与可读性：

复杂的业务流转、正则表达式、复杂的 SQL/ORM 查询必须添加中文单行或多行注释。

逻辑分支优先采用“提前返回 (Early Return)”原则，避免代码嵌套层级过深。
